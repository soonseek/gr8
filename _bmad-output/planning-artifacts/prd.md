## Executive Summary

### 제품 비전

**gr8**은 자동매매 시장의 투명성 혁명을 가져오는 탈중앙화 플랫폼입니다.

**핵심 비전:**
- **100% 오픈소스**: 모든 소스 코드가 공개되어 있어 누구나 LLM이나 개발자에게 작동 원리를 검증받을 수 있습니다
- **블록체인 기반 검증**: 전략의 백테스트 결과와 실행 성과가 변조 불가능한 블록체인에 기록됩니다
- **지식 공유 생태계**: "지식은 공유할수록 가치가 커진다"는 철학을 3계층 보상 시스템으로 실현합니다

**해결하는 문제:**

기존 자동매매 서비스들은 불투명한 백테스트 결과, 검증 불가능한 성과 주장, 그리고 사용자의 전략을 플랫폼이 소유하는 폐쇄적 구조를 가지고 있습니다. 사용자들은 "이 백테스트가 진짜야?"라는 불신과 "내 전략을 왜 너희가 가져가?"라는 종속감 속에서 서비스를 이용해야 합니다.

**타겟 사용자:**

1. **김준혁 (38세)**: "도박에서 탈출하고 싶은" 노련한 트레이더 - 삶을 되찾고 싶음
2. **이수민 (26세)**: "처음부터 제대로 배우고 싶은" 진지한 입문자 - 진입장벽 해소
3. **박지성 (34세)**: "지식이 자산이 되는" 전략 공유 크리에이터 - 수익화

### 무엇이 특별한가

**핵심 차별점: "지식은 공유할수록 가치가 커진다"는 철학을 블록체인으로 실현**

#### 3계층 보상 시스템 (산업 최초)

1. **직접 보상**: 전략을 공개하고 사용 시 스마트 컨트랙트가 자동으로 보상
2. **파생 보상**: 타인이 내 전략을 개선하면 수익의 일부를 원작자에게 자동 지급
3. **거버넌스**: 플랫폼 성장이 토큰 가치 상승으로 모든 참여자에게 간접 보상

#### 혁신적 통찰 (브레인스토밍 세션 발견)

**1. "진화-나눔-지식 생산"의 자립적 순환 구조**

gr8 사용자들은 "시스템 트레이딩은 진화가 필요하다"는 대전제에 동의합니다. 진화시키기 위해 나눠야 하고, 나누기 위해 지식을 생산해야 하는 순환 구조가 자립적으로 작동합니다.

**2. 실패 데이터의 순환 시스템 (학술 인용 모델)**

실패한 전략 데이터도 "실패 데이터 레이크"에 저장됩니다. Co-pilot Agent가 전략 검증 시 관련 실패 사례를 검색하고 인용하며, 인용 기록을 블록체인에 기록하고 원작자에게 자동 보상합니다. **"실패도 자산"** - 모든 것이 순환합니다.

**3. "전략이 유명해질수록 더 유효해진다"** (현물 시장)

시스템 트레이딩이 감정적 비효율성을 감소시키며, 전략이 유명해질수록 시장 안정화가 증대합니다. **전략 공유 = 시장 안정화 공공재 기여**

#### 랜딩 페이지/마케팅 경험

**중요:** 기술적, 기능적 구현과 더불어, **첫인상에서 고객을 확 끌어당길 수 있는 랜딩 페이지/홈 화면의 커뮤니케이션 구현**이 필수적입니다.

- **즉각적 신뢰 형성**: "이 백테스트 진짜야?" 불신을 해소하는 시각적 증거
- **명확한 가치 전달**: 5초 안에 "이게 뭐지? 왜 내가 필요로 하지?"를 이해시키는 메시징
- **감성적 연결**: "도박에서 탈출하고 싶은" 김준혁의 스토리를 보며 "이건 나야!"라고 느끼게 하는 스토리텔링

## Success Criteria

### 사용자 성공 (User Success)

#### 일반 사용자를 위한 4가지 핵심 성공 모멘트

**1. 첫 번째 백테스트 성공 🎯**
- **행동**: 노드-엣지 에디터로 첫 전략 생성 → 백테스트 실행
- **결과**: "+15% 수익! MDD -5%!" 확인
- **감정**: "와, 내가 코딩 없이 이걸 만들었다?"
- **지표**: 첫 주 차에 90% 이상의 사용자가 최소 1회 백테스트 완료

**2. 첫 실전 수익 💰**
- **행동**: 작은 금액으로 첫 실전 실행 (예: 10만 원)
- **결과**: 1주일 뒤 +3% 수익 확인
- **감정**: "이거 진짜 되네! 손매매와는 달라"
- **지표**: 첫 달 차에 60% 이상의 사용자가 실전 실행, 평균 +2% 이상 수익

**3. 첫 번째 수익화 💎**
- **행동**: 자신의 전략을 공개 (마켓플레이스에 퍼블리시)
- **결과**: 10명이 복제해서 50,000원 수익 획득
- **감정**: "내 지식이 돈이 되네!"
- **지표**: 3달 차에 20% 이상의 사용자가 자신의 전략 공개

**4. 생활 회복 🏞️**
- **행동**: 더 이상 실시간 차트를 보지 않음, 주말에 가족과 소풍
- **결과**: 도파민 중독에서 탈출, 삶의 질 개선
- **감정**: "이제 내 삶을 되찾았어"
- **지표**: 설문조사에서 80% 이상이 "삶의 질이 개선되었다"고 응답

#### 새로운 성공 모멘트

**5. 메타러닝의 순간 🧠**
- **성과**: "전략구성 방법론 자체에 대한 학습이 도구의 사용으로부터 발전"
- **예시**: 다른 사람의 전략을 보며 **"아, 이렇게 노드를 구성하면 되는구나!"** 깨달음
- **예시**: Co-pilot Agent가 리스크를 분석해주며 **"이런 패턴이 문제구나"** 학습
- **예시**: 실패 데이터 레이크 인용을 보며 **"아, 이건 피해야겠다"** 깨달음
- **지표**: 1달 사용 후 설문조사에서 70% 이상이 "트레이딩 사고방식이 발전했다" 응답

**6. 유튜버로서의 수익화 성공 🎬**
- **성과**: 자신의 전략을 공개하고 지속적인 수익 스트림 생성
- **감정**: "내 실력이 증명되고, 지속적으로 돈이 돌아와"
- **지표**: 공개한 전략이 평균 10회 이상 복제되어 월 수익 발생

#### Before & After 스토리텔링

**Before (도박에 빠진 삶):**
- "아침 9시부터 밤 12시까지 차트를 봄"
- "식사 중에도 폰으로 가격 확인"
- "자다가 일어나서 포지션 체크"
- "와이프: 당신 사람이야, 코인이야?"

**After (시스템 트레이더로서의 삶):**
- "출근할 때 전략 켜두고, 퇴근할 때 결과 보기"
- "주말에 가족과 소풍 🏞️"
- "내 전략이 공유돼서 월 100만 원 수익화"
- "이제 나도 도박꾼이 아니야, 시스템 트레이더야"

### 여정 1: 김준혁 - 삶을 되찾는 여정 (탈출을 원하는 노련한 트레이더)

아침 7시, 알람이 울리기도 전에 눈이 뜁니다. 출근 전에 차트를 한 번 더 확인해야죠. 회사에서는 업무에 집중이 안 됩니다. 점심시간에 매수하고, 퇴근 전에 매도합니다. 집에 오는 길에 지갑이 텁 비어있는 것 같습니다.

와이프와 저녁을 먹는데도 폰을 놓지 못합니다. "당신 사람이야, 코인이야?"라고 화냅니다. 김준혁은 "이번엔 정말 성공할 거야"라고 생각하지만, 속으로는 "이러다 다 잡는다"는 공포감에 시달립니다. **"이게 내 인생인가? 도박꾼으로 죽을 건가?"**

**Rising Action (발견과 시도):**

지갑 연결이 어렵습니다. Metamask가 처음이라 헷갈립니다. 하지만 튜토리얼이 친절해서 10분 만에 연결됩니다. 템플릿을 탐색합니다. "RSI + MACD 전략 - 6개월 동안 30% 수익, MDD -5%". "대박이네."

**Opening Scene (현재의 고통):**

이수민은 입사 1년 차 마케터로서, 점심시간에 동료들과 밥을 먹다가 코인 이야기를 자주 듣습니다. "비트코인 다시 올랐대", "이번에 NFT로 돈 버는 사람 있데" 이런 이야기들이죠.

싶지만... 두렵습니다. "손매매는 도박이래"라는 이야기를 들었어요. 친구 하나가 코인으로 3천만 원 벌었다가 다 날렸다고 합니다. 이수민은 **"처음부터 제대로 배우고 싶어"** 합니다.

집에 와서 유튜브를 봅니다. "코인 투자 방법", "비트코인 사는 법". 영상마다 다른 얘기를 합니다. "삼성전자 사서 5년 홀딩해라"는 사람도 있고, "이 코인 사세요"라는 사람도 있고... **"나는 멍청한가?"**라는 자책감이 듭니다.

TradingView에 가입해봅니다. 접속하자마자 **외계어**가 펼쳐집니다. RSI, MACD, 볼린저 밴드... 무슨 말인지 하나도 모르겠습니다. 10분 뒤 종료합니다. **"이건 내가 할 수 있는 게 아니야."**

**Rising Action (발견과 학습):**

친구가 gr8을 추천합니다. "진짜 투명하고, 노코딩이라서 쉬워."

gr8.com에 방문합니다. 랜딩 페이지에서 김준혁의 스토리를 봅니다.

> "Before: 아침 9시부터 밤 12시까지 차트를 봄"
> "After: 출근할 때 전략 켜두고, 퇴근할 때 결과 보기"

이수민은 **"이건 나야!"**라고 느낍니다. 바로 지갑 연결을 시도합니다. Metamask를 처음 써보지만, 튜토리얼이 따라하기 쉬워서 5분 만에 연결됩니다.

이제 뭘 해야 할지 모르겠습니다. "가이드" 버튼을 누릅니다. **"시작하기"** 튜토리얼이 시작됩니다:

1. **전략 둘러보기**: 템플릿 5개를 보여줍니다. "이건 RSI 전략이에요. 과매도/과매수를 판단해요."
2. **첫 번째 백테스트**: 템플릿을 선택하고 "백테스트" 클릭. 결과가 나옵니다.
3. **파라미터 조정**: "RSI 기간을 14에서 12로 바꿔볼까요?"

이수민은 **"드래그 앤 드롭이네? 쉽네!"**라고 느낍니다.

**Climax (성장과 깨달음):**

첫 주는 템플릿만 가지고 놀아봅니다. 10개의 백테스트를 돌려봅니다. 어떤 건 성공하고, 어떤 건 실패합니다. 하지만 중요한 건 **"내가 원인과 결과를 실험해보고 있다"**는 것입니다.

두 번째 주, 자신만의 전략을 만들어봅니다. RSI + MACD를 조합합니다. 노드를 연결하면서 **"아, 이렇게 생각하면 되는구나!"**라고 깨달음. 백테스트 결과: +12% 수익. **"와, 내가 코딩 없이 이걸 만들었다!"**

다른 사람의 전략도 보기 시작합니다. **"이 사람은 이렇게 생각했구나. 나는 이렇게 생각했어."** 비교하면서 배웁니다. 한 달 동안 20개의 전략을 만들고 백테스트합니다. 트레이딩 사고방식이 정말 발전했습니다.

**Resolution (자신감과 성장):**

3달 후, 이수민은 자신의 전략 2개를 공개합니다. 15명이 복제했습니다. 30,000원 수익이 떴습니다. **"내 지식이 돈이 되네!"** 큰 자랑은 아니지만, 뿌듯합니다.

친구들에게 자랑합니다. "나 자동매매 해. 내 전략으로." 친구들이 "대박, 가르쳐줘!"라고 합니다. 이수민은 **"나도 멋진 트레이더야"**라고 느낍니다.

친구 3명이 gr8에 가입했습니다. 이수민은 그들의 전략을 봐주고 조언해줍니다. **"나는 이제 초보자가 아니야."**

### 여정 3: 박지성 - 크리에이터의 수익화 여정 (전략 공유 크리에이터)

박지성은 7년 차 시스템 트레이더이자 프리랜서 개발자입니다. 이미 알파 트레이딩으로 2억을 벌었지만, **"이 지식이 더 가치 있게 쓰이면 좋겠어"**라고 생각합니다.

GitHub에 자신의 전략 코드를 오픈소스로 올립니다. 별로 쓰이지 않습니다. 누가 쓰더라도 **"이거 진짜야?"**라고 의심합니다. 3 Commas에 전략을 올려봅니다. 플랫폼 소유가 되어버립니다. "내 전략인데..."라는 박탈감.

Patreon을 만들어볼까요? 퀀트 펌 경력이 있지만, **"월회비 내고 내 전략을 써"**라고 설득하기 어렵습니다. 사람들은 "왜 돈 내고 써야 하지?"라고 묻습니다. 명확한 답이 없습니다.

박지성은 **"내 전략의 가치를 증명하고, 그 가치가 나에게 돌아오게 하고 싶어"** 합니다. 하지만 방법을 모르겠습니다.

**Rising Action (gr8과의 만남):**

박지성은 Hacker News에서 gr8을 발견합니다. "오픈소스 시스템 트레이딩 플랫폼... 3계층 보상 시스템... 블록체인 검증..."

관심이 갑니다.

gr8.com에 방문합니다. **"지식은 공유할수록 가치가 커진다"**는 철학을 봅니다. **"이건 내가 찾던 거야!"**

랜딩 페이지에서 **크리에이터 전용 안내**를 발견합니다:

> "당신의 전략을 수익화하세요."
> "전략 공개 시 자동 보상."
> "타인이 개선하면 수익 분배."
> "블록체인에 기록되어 증명 가능."

박지성은 **"이거야, 내 지식의 가치를 증명하고 돌아받는 구조!"**라고 느낍니다.

**크리에이터 온보딩:**

1. **크리에이터 신청**: 구독자 수/채널 입력 (박지성은 없음, 퀀트 경력으로 대체)
2. **전략 공개 가이드**: "어떻게 설명할지, 파라미터는 어떻게 문서화할지"
3. **가격 설정**: 무료 또는 유료 선택
4. **레퍼럴 링크**: 자신만의 링크 생성

박지성은 자신의 **"RSI Reversal Strategy"**를 공개합니다. 3년 동안 45% CARG, MDD -12%. 백테스트 결과와 함께 상세한 설명을 작성합니다. 가격은 10,000 G8D로 설정합니다.

**Climax (성공과 확장):**

일주일 뒤, 알림이 뜹니다. "누군가 당신의 전략을 복제했습니다." 그리고 "10,000 G8D 수익이 발생했습니다."

박지성은 기분이 좋습니다. **"내 전략이 진짜 가치가 있구나!"**

- **증권법 적용**: 일부 토큰/전략을 증권으로 분류 가능
- **Bank Secrecy Act**: 금융 기관으로서의 의무
- **Travel Rule**: $3,000 이상 거래 시 정보 공유 의무

**gr8 영향도:**
- ⚠️ 토큰/전략이 증권으로 분류되지 않도록 구조 설계
- ✅ Travel Rule 준수 프레임워크 구현
- ✅ $3,000 이상 거래 시 정보 수집 및 공유

#### 1.3 아시아 태평양

**한국 (특금법):**
- **VASP 라이선스**: 금융당국 등록 의무
- **자금세탁방지 (AML)**: 의심 거래 신고 의무
- **실명 확인**: 실명 인증 계정만 거래 가능

**일본 (FSA):**
- **등록 요건**: FSA 라이선스 취득
- **거래자 보호**: 자산 보유 규정

**홍콩 (SFC):**
- **규제 프레임워크**: 2026년까지 진행 예정
- **VASP 규정**: 라이선스 요건 강화

**gr8 영향도:**
- ✅ 다중 지역 규정 대응 가능한 아키텍처
- ✅ VASP 라이선스 취득 로드맵 필요
- ✅ 의심 거래 자동 감지 및 신고 시스템

### 2. 보안 아키텍처 (Security Architecture)

#### 2.1 Smart Contract Security

**2026년 표준:**
- **포괄적 감사**: 기관 투자자 요구 수준의 감사
- **온체인 서킷 브레이커**: 극심한 변동성 시 기능 일시 정지
- **크로스체인 보안 평가**: 수동 + 자동화 도구 결합
- **버그 바운티**: 공개 취약점 사냥 프로그램

**gr8 요구사항:**

**MVP Phase:**
- ✅ 스마트 컨트랙트 외부 감사 (MVP 전 완료)
- ✅ 펜트레이션 테스트 (취약점 0개, Critical)
- ✅ 온체인 서킷 브레이커 구현
- ✅ 가스 최적화 (트랜잭션 비용 < $0.10)

**Growth Phase:**
- ✅ 분기별 보안 감사
- ✅ 버그 바운티 프로그램 운영
- ✅ 보안 인시던트 대응 프로토콜
- ✅ 스마트 컨트랙트 업그레이드 메커니즘

**구체적 구현:**

```
[스마트 컨트랙트 보안 레이어]
├── 보상 분배 컨트랙트
│   ├── 재귀 탈취 방지 (Reentrancy Guard)
│   ├── 가스 한도 (Gas Limit per TX)
│   └── 일일 지급 한도 (Daily Payout Cap)
├── 전략 저장 컨트랙트
│   ├── 무결성 검증 (Integrity Check)
│   ├── 버전 관리 (Versioning)
│   └── 소유권 검증 (Ownership Verification)
└── 인용 기록 컨트랙트
    ├── 인용 카운팅 (Citation Counting)
    ├── 자동 보상 분배 (Auto Distribution)
    └── 조작 방지 (Tamper-Proof)
```

#### 2.2 백엔드 보안

**핵심 요구사항:**
- **데이터 암호화**: 저장 및 전송 시 AES-256
- **API 보안**: Rate limiting, Authentication, Input validation
- **인프라 보안**: VPC isolation, Firewall, DDoS protection

**구체적 구현:**

```
[백엔드 보안 레이어]
├── 데이터베이스
│   ├── RESTRICTED: 사용자 PII (KYC 데이터)
│   ├── CONFIDENTIAL: 거래 내역, 백테스트 결과
│   └── INTERNAL: 시스템 로그, 메트릭
├── API Gateway
│   ├── Rate Limiting (사용자/IP별)
│   ├── JWT Authentication
│   └── Request Validation
└── 인프라
    ├── VPC isolation (Public/Private subnet)
    ├── WAF (Web Application Firewall)
    └── DDoS protection (Cloudflare/ACF)
```

#### 2.3 프론트엔드 보안

**핵심 요구사항:**
- **XSS 방지**: Content Security Policy (CSP)
- **CSRF 방지**: Token-based validation
- **지갑 보안**: Secure communication with dApps

### 3. 감사 요구사항 (Audit Requirements)

#### 3.1 거래 투명성 및 추적성

**핵심 원칙:** "모든 거래는 추적 가능하고, 투명해야 한다"

**구현 요구사항:**

**1. 백테스트 결과 무결성:**
- ✅ 백테스트 실행 로그를 블록체인에 기록
- ✅ 사용 데이터 소스 해시값 저장
- ✅ 결과 조작 방지 (Immutable record)

**2. 전략 성과 추적:**
- ✅ 실전 실행 성과를 블록체인에 기록
- ✅ 수익 분배 내역 투명하게 공개
- ✅ 포크 및 개선 추적

**3. 보상 분배 감사:**
- ✅ 모든 보상 지급을 온체인에 기록
- ✅ 원작자 → 파생 보상 흐름 추적 가능
- ✅ 인용 기록 및 보상 분배 투명성

#### 3.2 규정 보고

**KYC/AML 보고:**
- ✅ 의심 거래 자동 감지 (STR - Suspicious Transaction Report)
- ✅ 고액 거래 보고 ($10,000 이상)
- ✅ 자금세탁 패턴 탐지

**데이터 보호 보고:**
- ✅ GDPR 준수 (EU 사용자)
- ✅ 개인정보 이용 내역 기록
- ✅ 데이터 침해 사고 보고 (72시간 이내)

#### 3.3 감사 로그

**기록 대상:**
- ✅ 모든 사용자 행동 (로그인, 전략 생성, 백테스트 실행)
- ✅ 모든 트랜잭션 (보상 지급, 전략 복제)
- ✅ 시스템 이벤트 (서버 시작/종료, 에러)
- ✅ 보안 이벤트 (로그인 실패, 비정상 행위)

**보관 정책:**
- ✅ 감사 로그: 최소 7년 보관 (금융 규정)
- ✅ 백업: 지리적으로 분리된 2개 이상의 지역에 저장

### 4. 사기 방지 (Fraud Prevention)

#### 4.1 실시간 이상 거래 탐지

**핵심 원칙:** "의심스러운 행위는 즉시 감지하고 차단한다"

**탐지 대상:**

**1. 백테스트 조작:**
- ✅ 비현실적 수익률 (+500% 이상)
- ✅ 의심스러운 데이터 소스 (API 호출 패턴 분석)
- ✅ 데이터 무결성 검증 (해시값 비교)

**2. 계정 이상 행위:**
- ✅ 동일 IP에서 다수 계정 생성
- ✅ 단 시간内 다수 전략 공개 (스팸)
- ✅ 반복적인 복제/취소 행위

**3. 스마트 컨트랙트 공격:**
- ✅ 재귀 탈취 공격 시도 감지
- ✅ 가스 소비량 이상 급증 (스팸 공격)
- ✅ 서비스 거부(DoS) 공격 탐지

#### 4.2 사기 방지 메커니즘

**자동 방지 시스템:**
- ✅ 실시간 모니터링 (이상 행태 분석)
- ✅ 자동 계정 정지 (의심 스코어 기반)
- ✅ 자동 전략 삭제 (조작 감지 시)

**수동 검토:**
- ✅ 운영자 대시보드에서 비정상 행위 확인
- ✅ 증거 수집 (로그, IP, 지갑 주소)
- ✅ 조사 후 영구 정지 또는 경고

**사후 대응:**
- ✅ 피해 사용자에게 보상
- ✅ 블랙리스트 등록 (지갑 주소, IP)
- ✅ 법적 조치 (필요 시)

#### 4.3 플래시 론 공격 방지

**DeFi 특화 보안:**

**1. 온체인 서킷 브레이커:**
- ✅ 극심한 변동성 시 스마트 컨트랙트 일시 정지
- ✅ 가스 가격 급등 시 트랜잭션 일시 중단
- ✅ 비정상적 거래량 급증 시 대기열 제한

**2. 가격 오라클 보안:**
- ✅ 다중 소스 가격 피딩 (Binance, Coinbase, Kraken)
- ✅ 시간 지연 가격 평균 (Flash price manipulation 방지)
- ✅ 가격 변동폭 제한 (±10% 이상 시 거래 중단)

**3. 유동성 관리:**
- ✅ 최소 유동성 요건 (저울 현상 방지)
- ✅ 거래량 제한 (단일 거래 당 최대액)
- ✅ 인출 한도 (일일/주간)

### 5. 데이터 보호 (Data Protection)

#### 5.1 개인정보 보호

**GDPR 준수 (EU):**
- ✅ 데이터 최소화 원칙 (최소한의 데이터만 수집)
- ✅ 사용자 동의 관리 (명시적 동의, 철회 권리)
- ✅ 데이터 접근 권한 (Right to access)
- ✅ 데이터 삭제 권한 (Right to be forgotten)
- ✅ 데이터 이동 권한 (Data portability)

**개인정보보호법 준수 (한국):**
- ✅ 개인정보 수집/이용 동의
- ✅ 제3자 제공 동의
- ✅ 영업 비밀 보호
- ✅ 정보통신망 이용촉진 및 정보보호 등에 관한 법률 준수

#### 5.2 데이터 암호화

**저장 데이터 암호화:**
- ✅ 사용자 PII: AES-256 암호화
- ✅ 지갑 주소: Public key만 저장
- ✅ 백테스트 결과: 무결성 해시값 + 암호화

**전송 데이터 암호화:**
- ✅ HTTPS/TLS 1.3 (모든 API 통신)
- ✅ WebSocket Secure (WSS)
- ✅ 지갑 통신: EIP-191 Signed data

#### 5.3 데이터 접근 제어

**권한 관리:**

```
[데이터 접근 권한 레벨]
├── Level 1: Public (모든 사용자)
│   └── 공개 전략, 백테스트 결과, 크리에이터 프로필
├── Level 2: Authenticated User (인증된 사용자)
│   └── 자신의 전략, 수익 대시보드, 거래 내역
├── Level 3: Creator (크리에이터)
│   └── 자신의 전략 수익, 레퍼럴 데이터, 팬 demographics
└── Level 4: Admin (운영자)
    └── 모든 사용자 데이터, 시스템 로그, 보안 대시보드
```

**접근 제어:**
- ✅ JWT 토큰 기반 인증
- ✅ RBAC (Role-Based Access Control)
- ✅ 최소 권한 원칙 (Principle of Least Privilege)
- ✅ 접속 로그 기록 (누가, 언제, 무엇을)

### 발견된 혁신 영역

gr8은 자동매매 시장의 폐쇄성을 타파하고 진정한 투명성 혁명을 가져오는 **4가지 핵심 혁신**을 도입합니다.

#### 혁신 1: 3계층 보상 시스템 (Industry First)

**문제:**
기존 자동매매 플랫폼(Mudrex, 3Commas, Cryptohopper 등)에서 전략을 공유해도:
- 플랫폼이 소유권을 가져감
- 일회성 수수료만 지급
- 타인이 개선해도 원작자는 보상받지 못함

**혁신적 해결:**
gr8은 **산업 최초**의 3계층 보상 시스템을 도입합니다:

1. **직접 보상**: 전략 복제 시 스마트 컨트랙트가 자동 지급
2. **파생 보상**: 타인이 내 전략을 개선하면 수익의 일부를 원작자에게 자동 지급
3. **거버넌스**: 플랫폼 성장이 토큰 가치 상승으로 모든 참여자에게 간접 보상

**시장 차별점:**
- 기존: "공유하면 플랫폼 소유" → gr8: "공유해도 내 소유, 지속적 수익"
- 기존: "일회성 수수료" → gr8: "파생 보상으로 무한 수익"
- 기존: "블랙박스 성과" → gr8: "블록체인에 기록된 검증 가능한 성과"

**Sources:**

#### 혁신 2: 실패 데이터의 순환 시스템 (Academic Citation Model)

**문제:**
시스템 트레이딩에서 "실패한 전략"은 보통 삭제되거나 잊혀집니다. 하지만 이 실패 데이터에는 귀중한 교훈이 포함되어 있습니다.

**혁신적 해결:**
gr8은 **학술 논문의 인용 모델**을 실패 데이터에 적용합니다:

**실행 프로세스:**
1. 실패한 전략을 "실패 데이터 레이크"에 저장 (사용자 동의 하에)
2. Co-pilot Agent가 전략 검증 시 관련 실패 사례를 자동 검색
3. 실패 데이터를 인용하고, 인용 기록을 블록체인에 기록
4. 원작자에게 자동 보상 지급 ("실패도 자산")

**가치 제안:**
- **원작자**: "실패한 전략도 수익화" (새로운 수익 스트림)
- **사용자**: "다른 사람의 실패를 배워서 시간 절약"
- **생태계**: "지식이 순환하며 모두가 발전"

**시장 차별점:**
- 기존: "실패는 삭제" → gr8: "실패도 인용하고 보상"
- 이 모델을 적용한 플랫폼은 **시장에 없음**

**Sources:**
- [Knowledge Sharing Platforms: An Empirical Study](https://www.mdpi.org/2071-1050/12/6/2341)
- [Large language models reduce public knowledge sharing](https://academic.oup.com/pnasnexus/article/3/9/pgae400/7754871)

#### 혁신 3: 모든 시장 개방 + 시장 효율화 기여 (Market Agnostic)

**문제:**
기존 플랫폼은 특정 시장에 최적화되어 있고, 시장 제한이 있습니다.

**혁신적 해결:**
gr8은 **모든 시장(현물, 선물, 옵션, 퍼페추얼, 마진)을 개방**합니다:

**지원 시장:**
- ✅ 현물 (Spot)
- ✅ 선물 (Futures)
- ✅ 옵션 (Options)
- ✅ 퍼페추얼 (Perpetual)
- ✅ 마진 (Margin)

**혁신적 가치:**

**1. 시장 불변 (Market Agnostic) 플랫폼**
- 기존: 특정 시장에 최적화된 플랫폼 (예: 현물만)
- gr8: **모든 시장을 지원하는 통합 플랫폼**
- 사용자는 시장 선택의 자유를 완전히 가짐

**2. 시장 효율화에 기여**
- 현물: 전략 공유 = 시장 안정화 (감정적 비효율성 감소)
- 선물/옵션: 전략 공유 = 가격 발견 개선 + 유동성 증가
- **결과**: 모든 시장의 효율성이 향상되는 공공재 기여

**3. 솔직한 커뮤니케이션**
- "전략이 유명해질수록 수익이 줄어들 수도 있습니다" (솔직함)
- 하지만 "시장 전체의 효율성은 향상됩니다" (공공재 기여)
- 그리고 "gr8의 보상 시스템으로 수익화는 지속됩니다" (수익화 보장)

**시장 차별점:**
- 기존: 특정 시장에 특화 (예: 현물 봇, 선물 봇 따로)
- gr8: **모든 시장을 통합한 단일 플랫폼**
- 사용자는 전략을 복사해서 다른 시장에도 적용 가능

#### 혁신 4: 100% 오픈소스 + 블록체인 검증

**문제:**
기존 플랫폼의 백테스트는 불투명합니다:
- "이 백테스트 진짜야?"라는 불신
- 조작 가능한 결과
- 검증 불가능한 성과 주장

**혁신적 해결:**
gr8은 **100% 오픈소스 + 블록체인 검증**으로 완전한 투명성을 제공합니다:

**실행:**
- 모든 소스 코드 공개 (GitHub)
- 백테스트 결과를 블록체인에 기록 (변조 불가능)
- 사용자가 LLM/개발자에게 작동 원리 검증 가능

**시장 차별점:**
- 기존: "블랙박스" → gr8: "투명한 백스"
- 검증 가능한 투명성을 제공하는 플랫폼은 **시장에 없음**

### 검증 접근 방식 (Validation Approach)

#### 1. 백테스트 투명성 검증

**방법:**
- 백테스트 실행 로그를 블록체인에 기록
- 데이터 소스 해시값 저장
- 사용자가 독립적으로 재검증 가능

**성공 기준:**
- 사용자 설문조사: "백테스트 투명성" 신뢰도 70% 이상
- 외부 감사: 백테스트 무결성 확인

#### 2. 3계층 보상 시스템 검증

**방법:**
- MVP Phase: 직접 보상만 구현 (가장 간단)
- Growth Phase: 파생 보상 추가
- Vision Phase: 거버넌스 추가

**성공 기준:**
- 3개월 차: 최소 20% 사용자가 자신의 전략 공개
- 6개월 차: 포크 대 원본 비율 0.5:1 달성

#### 3. 실패 데이터 순환 시스템 검증

**방법:**
- Phase 2 (Growth): 실패 데이터 레이크 구현
- 사용자 동의 하에 실패 전략 저장
- Co-pilot Agent 베타에서 인용 시스템 테스트

**성공 기준:**
- 실패 데이터 인용 횟수: 월 50회 이상
- 원작자 보상: 월 평균 10명 이상의 실패 데이터 작성자가 수익

#### 4. 유튜버/인플루언서 수익화 검증

**방법:**
- MVP: 유튜버 전용 온보딩 + 레퍼럴 시스템
- 레퍼럴 추적 정확도: 100% (블록체인 기록)

**성공 기준:**
- 3개월 차: 최소 3명의 유튜버가 전략 30회 이상 공유
- 유튜버 평균 수익: 기존 레퍼럴 대비 2배 이상

### 위험 완화 (Risk Mitigation)

#### 위험 1: 사용자가 실패 데이터 공유를 거부할 수 있음

**완화:**
- 옵트인(Opt-in) 모델: 실패 데이터 공유는 선택사항
- 명확한 가치 전달: "실패도 수익화" 혜택 강조
- 프라이버시 보호: 익명화 옵션 제공

**대체안:**
- 공유하지 않아도 기본 기능 사용 가능

#### 위험 2: 3계층 보상 시스템의 복잡성

**완화:**
- 단계적 롤아웃: MVP → Growth → Vision
- 직관적인 UI/UX: 보상 흐름 시각화
- 명확한 문서화: 어떻게 보상이 되는지 설명

**대체안:**
- MVP Phase: 직접 보상만 (단순화)
- 복잡도가 높으면 파생 보상 지연

#### 위험 3: 블록체인 검증의 비용

**완화:**
- 가스 최적화: 트랜잭션 비용 <$0.10
- 배치 처리: 여러 백테스트를 하나의 트랜잭션에 기록
- L2 솔루션 고려 (거래 비용 절감)

**대체안:**
- MVP Phase: 백테스트 결과는 중앙화 DB + 해시값만 온체인
- 사용자 성장 후 전체 온체인

#### 위험 4: 선물/옵션 시장에서 전략이 유명해지면 알파 감소

**완화:**
- 솔직한 커뮤니케이션: "전략이 유명해지면 수익이 줄어들 수도 있습니다"
- 하지만 "시장 효율성에는 기여합니다" (공공재 가치)
- 그리고 "gr8의 보상 시스템으로 수익화는 지속됩니다"

**대체안:**
- 데이터로 검증: 전략 공유 횟수 vs 수익률 상관관계 분석
- 가정이 틀리면 솔직하게 인정하고 피벗

#### 위험 5: 유튜버가 새로운 수익 모델을 받아들이지 않을 수 있음

**완화:**
- 프루프 오브 컨셉(PoC): 소수 유튜버와 베타 테스트
- 비교 우위 강조: "일회성 레퍼럴 vs 지속적 수익"
- 성공 사례 확산: 초기 유튜버 성공 후 홍보

**대체안:**
- 일반 크리에이터(유튜버 아님) 타겟팅
- 거래소 레퍼럴과 병행 지원

### 가스 최적화 (Gas Optimization)

#### 최적화 목표

**타겟:** 트랜잭션 비용 **<$0.10** (사용자 기준)

#### 최적화 전략

**1. 배치 처리 (Batching)**
- 여러 백테스트 결과를 하나의 트랜잭션에 기록
- 일일 보상 분배를 배치로 처리
- 효과: 가스 비용 70-90% 절감

**2. State Variable 최적화**
- `uint256` 대신 필요한 최소 크기 사용 (uint8, uint16)
- `bool` 대신 `uint8(0/1)` 고려 (가끔 더 저렴)
- packing: 여러 작은 변수를 하나의 slot에 저장

**3. Storage vs Memory**
- 불필요한 storage 읽기/쓰기 회피
- Memory는 storage보다 훨씬 저렴

**4. Event 로그 최적화**
- 중요한 데이터만 Event로 기록
- 인덱싱된 파라미터 최소화 (indexed는 가스 비용 증가)

**5. Lazy Minting**
- 필요한 시점에만 트랜잭션 실행
- 사용자가 명시적 요청 시만 온체인 기록

**구체적 예시:**

```solidity
// 비효율적 (가스 비용 높음)
function recordBacktest(string memory result) public {
    backtests.push(result); // storage 쓰기
}

// 효율적 (가스 비용 낮음)
function recordBacktestBatch(string[] memory results) public {
    for (uint i = 0; i < results.length; i++) {
        emit BacktestRecorded(results[i]); // event만 기록
    }
}
```

**L2 솔루션 고려 (Phase 2):**
- Polygon, Arbitrum, Optimism
- 효과: 가스 비용 95-99% 절감
- 단점: 추가 브리지 비용, 보안 모델 변경

- **전략:** MVP에서는 기본 기능만, Growth에서 확장

> "가장 빠르게 핵심 가치(투명성)를 증명하고, 유튜버 타겟으로 수익화를 검증한다."

**리소스 요구사항 (MVP):**
- **팀 규모:** 4-6명
  - 프론트엔드: 1-2명 (React + Web3)
  - 백엔드: 1명 (Node.js + ethers.js)
  - 스마트 컨트랙트: 1명 (Solidity)
  - 디자이너: 1명
  - PM/Founders: 1-2명

- **기술 스택:**
  - Frontend: Next.js + RainbowKit + wagmi + viem
  - Backend: Node.js + ethers.js + PostgreSQL
  - Smart Contracts: Solidity + Hardhat
  - Blockchain: Monad L1
  - Infrastructure: Vercel (Frontend) + AWS/GCP (Backend)

- **개발 기간:**
  - Smart Contract 개발 + 감사: 2-3개월
  - MVP 개발: 3-4개월
  - **총: 5-7개월** (감사 병렬 진행 시)

### 위험 완화 전략

#### 기술적 위험

**가장 기술적으로 어려운 부분:**
1. 스마트 컨트랙트 보안 (재귀 탈취, 악의적 공격)
2. 백테스팅 엔진 정확도 (99.9% 무결성)
3. 노코드 에디터 복잡도 (n8n 스타일)

**초기 구현 단순화:**
- ✅ MVP: 스마트 컨트랙트 기능 최소화 (직접 보상만)
- ✅ MVP: 백테스팅 엔진 = Binance API + 단순 계산
- ✅ MVP: 노코드 에디터 = 기본 노드만 (5-10개)

**기술 타당성 가정:**
- ✅ Monad L1이 2026년에 메인넷으로 안정화될 것이다
- ✅ Binance API가 안정적일 것이다
- ✅ 스마트 컨트랙트 감사가 Critical 0개로 가능할 것이다

**완화 전략:**
- MVP 전에 감사 완료 (전제 조건)
- 펜트레이션 테스트 병행
- 백업 체인 고려 (Ethereum or Polygon)

#### 시장 위험

**가장 큰 시장 위험:**
1. 사용자가 "블록체인 검증"을 중요하게 생각하지 않을 수도 있음
2. 유튜버가 새로운 수익 모델을 받아들이지 않을 수도 있음
3. 경쟁사가 빠르게 따라 할 수 있음

**MVP가 이 위험을 어떻게 해결하는가:**
1. **투명성 불신 해결**: 랜딩 페이지에서 강력한 스토리텔링 ("이 백테스트 진짜야?")
2. **유튜버 검증**: 3명의 유튜버와 PoC 실행 (30회 이상 공유 목표)
3. **선발자 우위**: 100% 오픈소스로 생태계 확장

**필요한 학습:**
- 유튜버 인터뷰: "레퍼럴보다 지속적 수익이 더 매력적인가?"
- 사용자 테스트: "블록체인 검증을 신뢰하는가?"
- 경쟁사 분석: 빠른 복제 방어 방안

#### 리소스 위험

**계획보다 리소스가 부족할 경우:**
- **최소 팀 규모:** 3명 (프론트엔드 + 백엔드 + 스마트 컨트랙트)
- **절감 가능한 기능:**
  - 노코드 에디터 복잡도 감소 (기본 노드만)
  - 템플릿 라이브러리 축소 (5개만)
  - 유튜버 전용 기능 지연 (일반 사용자 먼저)

**더 작은 기능 세트로 출시 가능:**
- ✅ "백테스트만 가능한 플랫폼"
  - 전략 공유 기능 제외
  - 수익화 제외
  - 유튜버 타겟팅 제외
  - 핵심 가치(투명성)만 증명

**리소스 완화:**
- MVP 개발 6개월 후 출시 (속도보다 정확성)
- 외부 감사비용 예산 확보 ($50,000+)
- Emergency fund (예상치 못한 이슈 대응)

## Non-Functional Requirements

> NFR은 시스템이 **얼마나 잘** 수행되어야 하는지를 정의합니다. 각 NFR은 구체적이고 측정 가능해야 합니다.
>
> 선택적 접근: 이 제품에 실제로 중요한 NFR 카테고리만 포함합니다. 불필요한 요구사항 팽창을 방지하고 실제로 중요한 것에 집중합니다.

---

### 1. Performance (성능)

사용자 경험에 직접적인 영향을 미치는 성능 요구사항:

- **NFR-PERF-001**: 시스템은 90th percentile 백테스트 결과를 30초 이내에 제공해야 한다
- **NFR-PERF-002**: 시스템은 모든 API 요청의 95th percentile 응답 시간을 200ms 이내로 처리해야 한다
- **NFR-PERF-003**: 시스템은 페이지 로드를 2초 이내에 완료해야 한다 (초방 방문자)
- **NFR-PERF-004**: 시스템은 지갑 연결을 10초 이내에 완료해야 한다
- **NFR-PERF-005**: 시스템은 전략 복제를 3초 이내에 완료해야 한다

### 2. Security (보안)

핀테크 HIGH 복잡도 도메인의 보안 요구사항:

- **NFR-SEC-001**: 시스템은 모든 저장 데이터를 AES-256으로 암호화해야 한다 (At Rest)
- **NFR-SEC-002**: 시스템은 모든 전송 데이터를 TLS 1.3으로 암호화해야 한다 (In Transit)
- **NFR-SEC-003**: 시스템은 스마트 컨트랙트 배포 전 외부 보안 감사를 완료해야 한다 (Critical 0개)
- **NFR-SEC-004**: 시스템은 펜트레이션 테스트를 완료해야 한다 (Critical 취약점 0개)
- **NFR-SEC-005**: 시스템은 모든 사용자 PII(개인식별정보)에 대한 접근을 제어해야 한다 (RBAC)
- **NFR-SEC-006**: 시스템은 Reentrancy 공격을 방지하기 위해 모든 스마트 컨트랙트에 보호 메커니즘을 구현해야 한다
- **NFR-SEC-007**: 시스템은 온체인 서킷 브레이커를 구현하여 극심한 변동성 시 스마트 컨트랙트를 일시 정지할 수 있어야 한다
- **NFR-SEC-008**: 시스템은 모든 트랜잭션과 사용자 행동을 감사 로그로 기록해야 한다 (7년 보관)
- **NFR-SEC-009**: 시스템은 백테스트 결과의 무결성을 보장하기 위해 데이터 소스 해시값을 저장해야 한다
- **NFR-SEC-010**: 시스템은 비정상 행위(백테스트 조작, 계정 이상 행위)를 실시간으로 감지해야 한다

**규제 준수:**
- **NFR-SEC-011**: 시스템은 KYC/AML 프로세스를 지원해야 한다 (3단계: Selfie, ID, 주소지)
- **NFR-SEC-012**: 시스템은 의심 거래를 자동 감지하고 신고할 수 있어야 한다
- **NFR-SEC-013**: 시스템은 $3,000 이상 거래 시 Travel Rule을 준수해야 한다

### 3. Scalability (확장성)

유튜버 트래픽 스파이크와 사용자 성장을 지원하는 확장성:

- **NFR-SCALE-001**: 시스템은 동시에 1,000명의 활성 사용자를 지원할 수 있어야 한다 (MVP 목표)
- **NFR-SCALE-002**: 시스템은 3개월 차에 10,000명의 활성 사용자로 확장할 수 있어야 한다 (성장 목표)
- **NFR-SCALE-003**: 시스템은 트래픽 스파이크(유명 유튜버 영상 게시)를 처리하기 위해 오토스케일링을 지원해야 한다
- **NFR-SCALE-004**: 시스템은 10배 사용자 성장 시 성능 저하를 10% 이내로 유지해야 한다
- **NFR-SCALE-005**: 시스템은 백테스팅 큐를 처리하기 위해 수평적 확장이 가능해야 한다
- **NFR-SCALE-006**: 시스템은 데이터베이스 읽기 병목을 방지하기 위해 리플리카를 지원해야 한다

### 4. Reliability (신뢰성)

시스템 가용성과 안정성 요구사항:

- **NFR-REL-001**: 시스템은 99% 이상의 가용성을 제공해야 한다 (월간 기준)
- **NFR-REL-002**: 시스템은 계획된 중단 시 24시간 전에 사용자에게 통지해야 한다
- **NFR-REL-003**: 시스템은 비계획된 중단 시 1시간 이내에 복구해야 한다 (MTTR)
- **NFR-REL-004**: 시스템은 데이터 손실률 0%를 유지해야 한다 (백테스트 결과, 사용자 데이터)
- **NFR-REL-005**: 시스템은 지리적으로 분리된 2개 이상의 리전에 백업을 저장해야 한다
- **NFR-REL-006**: 시스템은 일일 백업을 자동으로 수행해야 한다

---

### 5. Integration (통합)

외부 시스템 연동 요구사항:

- **NFR-INT-001**: 시스템은 Binance API를 통해 히스토리컬 데이터를 조회할 수 있어야 한다
- **NFR-INT-002**: 시스템은 Binance API 장애 시 99.9% 가용성을 유지해야 한다 (백업 API)
- **NFR-INT-003**: 시스템은 MetaMask 지갑과 Web3 표준(EIP-191)으로 통신해야 한다
- **NFR-INT-004**: 시스템은 WalletConnect 프로토콜을 통해 100+ 지갑과 연동할 수 있어야 한다
- **NFR-INT-005**: 시스템은 YouTube API를 통해 채널 정보를 검증할 수 있어야 한다
- **NFR-INT-006**: 시스템은 Monad L1 블록체인과 RPC 통신을 할 수 있어야 한다
- **NFR-INT-007**: 시스템은 외부 API 장애 시 회로탄적(fallback) 메커니즘을 구현해야 한다

### 6. Blockchain/Web3 (블록체인)

Project-Type 특화 Web3 요구사항:

- **NFR-BC-001**: 시스템은 트랜잭션당 가스 비용을 $0.10 미만으로 유지해야 한다
- **NFR-BC-002**: 시스템은 온체인 트랜잭션 지연 시간을 30초 이내로 처리해야 한다
- **NFR-BC-003**: 시스템은 지갑 연결 성공률을 95% 이상 유지해야 한다
- **NFR-BC-004**: 시스템은 가스 가격 변동성에 대해 예측 가능한 가스비를 제공해야 한다
- **NFR-BC-005**: 시스템은 배치 처리를 통해 가스 비용을 최적화해야 한다 (여러 백테스트를 하나의 트랜잭션에)
- **NFR-BC-006**: 시스템은 잘못된 체인 연결 시 자동으로 올바른 체인으로 전환을 요청해야 한다
